# CMP 后端服务落地方案

## 1. 架构选择

建议采用云原生模块化架构。落地方式有两种：

- 阶段一：模块化单体，加清晰领域模块、独立 schema、异步消息和接口边界。
- 阶段二：按领域拆分微服务，将高吞吐和高变更模块优先独立，例如 CDR、Rating、Billing、Supplier Integration、eSIM Orchestration。

如果团队一开始人数有限，推荐从模块化单体起步，但设计时严格遵守服务边界，避免后续拆分困难。

## 2. 推荐服务清单

### API Gateway

职责：

- 路由转发。
- OAuth/JWT 校验。
- WAF。
- 限流。
- 请求日志。
- correlation_id 注入。
- OpenAPI 文档入口。

### IAM Service

职责：

- 用户、角色、权限。
- SSO/OIDC/SAML。
- API Client。
- Token 签发和校验。
- MFA。
- SCIM 扩展。

### Account Service

职责：

- 账户树。
- Reseller 层级。
- 账户额度。
- 账户套餐授权。
- Billing Profile 基础资料。
- 数据范围判断。
- 账户状态机：draft、pending_review、active、suspended、closed。
- 风控和财务限制：credit_hold、compliance_hold、fraud_hold、overdue、bad_debt。

### SIM Inventory Service

职责：

- SIM/码号库存。
- 库存状态和服务状态。
- SIM 分组。
- 网络策略。
- SIM 诊断数据汇总。
- 激活、暂停、恢复、终止状态机。

### eSIM Orchestration Service

职责：

- eID/eUICC 管理。
- Profile 库存。
- eSIM 操作状态机。
- RSP 流程编排。
- 与 Supplier Integration 协作调用 SM-DP+/eIM。

### Supplier Integration Service

职责：

- 供应商适配器。
- 鉴权配置。
- 产品同步。
- 开通、暂停、恢复、销户调用。
- CDR 拉取。
- API 调用日志和 SLA。

### Product Package Service

职责：

- 套餐。
- Rate Plan 版本。
- 地区 Zone。
- 流量池规则。
- 套餐授权。

### CDR Ingestion Service

职责：

- 原始 CDR 文件接收。
- 文件校验。
- 供应商 Parser。
- 标准化 CDR。
- 去重。
- 解析失败处理。

### Rating Service

职责：

- 用量计费。
- 流量池扣减。
- 超额计费。
- 阶梯计费。
- Rate Plan 版本匹配。
- 生成 Rated Usage。

### Billing Service

职责：

- 账单周期。
- 账单运行。
- 账单预览。
- 账单发布。
- 调整项。
- 发票 PDF。
- 邮件发送。
- Reseller 结算。

### Batch Job Service

职责：

- 文件上传。
- 模板管理。
- 结构校验。
- 业务校验。
- 审批。

### Notification Service

### Analytics Service

### Audit Service

- 操作追踪。

## 3. 服务间通信

同步调用：

- API Gateway 到业务服务。
- 后台详情查询。
- 权限校验。
- 状态查询。

异步事件：

- `sim.operation.requested`
- `sim.operation.completed`
- `esim.operation.requested`
- `esim.operation.completed`
- `cdr.file.received`
- `cdr.normalized`
- `usage.rated`
- `invoice.generated`
- `invoice.issued`
- `batch_job.completed`
- `webhook.delivery.failed`

消息中间件：

- Kafka/Pulsar：适合高吞吐 CDR 和事件流。
- RabbitMQ：适合任务队列和工作流。
- Redis Stream：适合初期轻量异步，但长期建议使用更标准的 MQ。

## 4. eSIM 操作落地

eSIM 操作必须采用状态机和工作流，不建议同步阻塞等待供应商最终结果。

状态：

- accepted
- validating
- submitted
- processing
- succeeded
- failed
- timeout
- cancelled

流程：

1. API 接收请求。
2. 校验权限、账户范围、Profile 状态和套餐状态。
3. 创建 esim_operations，写入 idempotency_key。
4. 发送 `esim.operation.requested`。
5. Worker 调用 Supplier Adapter。
6. 记录 supplier_transaction_id。
7. 供应商回调或轮询更新状态。
8. 状态机更新 Profile/SIM 状态。
9. 发送 Webhook 和审计日志。

幂等：

- 相同 Profile、operation_type、idempotency_key 只能创建一次操作。
- 如果供应商接口不可确认，进入 unknown/processing，由补偿任务查询。

## 5. 号码服务状态机落地

号码/SIM 的激活、暂停、恢复、终止必须由 SIM Inventory Service 统一编排，不能由 Controller、批量任务或供应商回调直接修改最终状态。

### 5.1 状态字段

建议将 SIM 状态拆成两个字段：

- inventory_status：stock、reserved、assigned、recycled、retired。
- service_status：not_started、pending_activation、test_ready、active、suspension_pending、suspended、resume_pending、termination_pending、terminated、failed。

这样可以区分“号码归属库存状态”和“网络服务可用状态”。例如一个 SIM 可以 assigned 但 not_started，也可以 assigned 且 suspended。

### 5.2 操作执行流程

激活、暂停、恢复、终止统一走以下流程：

1. API Gateway 接收请求并校验 Token。
2. IAM 校验操作权限和账户范围。
3. SIM Inventory Service 检查账户状态、SIM 状态、套餐状态和是否存在未完成操作。
4. 对高风险操作执行 preview，必要时创建审批单。
5. 创建 sim_operations 记录，写入 idempotency_key 和 correlation_id。
6. 状态机将 SIM 切换到 pending 状态，例如 pending_activation 或 suspension_pending。
7. 发布领域事件，例如 `sim.operation.requested`。
8. Worker 调用 Supplier Integration Service。
9. Supplier Adapter 返回 success、failed、accepted 或 unknown。
10. 状态机根据供应商最终结果更新 service_status。
11. 发布 Webhook、通知和审计日志。

### 5.3 操作前置条件

激活：

- inventory_status 为 reserved 或 assigned。
- service_status 为 not_started 或 failed。
- 账户为 active，且不存在阻断型 risk_status。
- 套餐 active，账户有套餐授权。
- 供应商资源和接口健康。

暂停：

- service_status 为 active。
- 不存在其他进行中 sim_operation。
- 操作者拥有 `sim.suspend` 权限。
- 批量暂停超过阈值时需要审批。

恢复：

- service_status 为 suspended。
- 账户不处于 credit_hold、compliance_hold、fraud_hold。
- 套餐未过期，或恢复策略允许重新订阅。
- 供应商支持恢复。

终止：

- service_status 为 active 或 suspended。
- 通过费用和影响预览。
- 批量终止、重要客户或存在未出账用量时需要审批。
- 终止成功后不可直接恢复，只能重新开通新订阅。

### 5.4 副作用处理

激活成功：

- SIM service_status 改为 active。
- subscription 改为 active。
- 设置 activated_at。
- 创建 billing anchor。
- 发送 `sim.activated`。

暂停成功：

- SIM service_status 改为 suspended。
- 设置 suspended_at 和 suspension_reason。
- 根据暂停策略决定 subscription 是继续计费、暂停计费还是账期保持。
- 发送 `sim.suspended`。

恢复成功：

- SIM service_status 改为 active。
- 清除暂停原因。
- 恢复 subscription 或重新校验账期。
- 发送 `sim.resumed`。

终止成功：

- SIM service_status 改为 terminated。
- subscription 改为 terminated。
- 设置 terminated_at。
- 触发最终计费和资源释放。
- 发送 `sim.terminated`。

### 5.5 一致性要求

- 状态切换、operation 更新、领域事件写入建议使用 Outbox Pattern。
- 同一 SIM 同一时间只能有一个未完成服务操作。
- 供应商结果 unknown 时进入补偿查询，不直接当作失败。
- 所有状态切换必须写 audit_logs。
- 批量任务逐行独立执行，默认部分成功、部分失败，除非任务指定 all_or_nothing。

## 6. CDR 处理落地

CDR 是平台高吞吐核心链路。

流程：

1. Supplier Adapter 拉取或接收 CDR 文件。
2. 原始文件上传对象存储。
3. 创建 raw_cdr_files 记录。
4. Parser Worker 按供应商格式解析。
5. 生成标准 usage_cdrs。
6. 根据 raw_record_hash 和 supplier_record_id 去重。
7. 发布 `cdr.normalized`。
8. Rating Service 消费并生成 rated_usage。
9. Analytics Service 更新聚合表。
10. Billing Service 在账期内消费 rated_usage。

异常处理：

- 文件级失败：标记 raw_cdr_files failed。
- 行级失败：写入 cdr_parse_errors。
- 重复记录：标记 duplicate，不进入重复计费。
- Parser 版本变更：保留 parser_version。

## 7. Rating 落地

Rating 引擎输入：

- usage_cdr
- subscription
- package
- rate_plan version
- usage_pool
- account contract override

输出：

- rated_usage
- usage_pool deduction
- threshold events
- overage events

规则：

- 每次计费记录 rate_plan_id 和 version。
- 计费过程必须可重放，但已出账数据不能直接覆盖。
- 重跑 Rating 只能产生差异调整或新的版本结果，由 Billing 决定如何处理。

## 8. Billing 落地

账单运行流程：

1. 创建 invoice_run。
2. 锁定账户和账期。
3. 拉取未出账 rated_usage。
4. 计算周期费、超额费、折扣、税费、调整项。
5. 生成 draft invoice。
6. 财务预览和审批。
7. 发布 invoice。
8. 生成 PDF 和明细文件。
9. 邮件发送。
10. 标记 rated_usage 已归属 invoice。

账单不可变原则：

- issued 后不允许直接改金额。
- 错误通过 credit note、debit note 或 adjustment invoice 处理。

## 9. 批量任务落地

批量任务采用统一框架。

通用流程：

1. 上传文件到对象存储。
2. 创建 batch_job。
3. 按模板解析。
4. 写入 batch_job_items。
5. 结构校验。
6. 业务校验。
7. 等待审批。
8. Worker 执行。
9. 每行写结果。
10. 汇总任务状态。

不同任务通过 handler 扩展：

- SimImportHandler
- EsimProfileImportHandler
- PackageChangeHandler
- EsimOperationHandler
- CdrReprocessHandler
- AccountImportHandler

## 10. 数据库与存储

推荐：

- PostgreSQL：核心事务数据。
- TimescaleDB 或 PostgreSQL 分区表：用量和 CDR。
- ClickHouse/BigQuery/Snowflake：长期分析和大规模报表，可作为二期。
- Redis：缓存、分布式锁、幂等结果、短期计数。
- Object Storage：原始 CDR、导出文件、账单 PDF、批量文件。
- OpenSearch/Elasticsearch：日志检索和 SIM 快速搜索，可选。

分区建议：

- usage_cdrs 按月分区，再按 supplier/account 建索引。
- rated_usage 按账期分区。
- audit_logs 按月分区。
- supplier_api_logs 按月分区。

## 11. 缓存策略

适合缓存：

- 当前用户权限。
- 账户树摘要。
- 套餐可见性。
- Dashboard 聚合结果。
- API Client 限流计数。

不适合直接缓存：

- 账单最终金额。
- eSIM 最新状态，除非有明确 TTL 和刷新策略。
- 供应商敏感凭证明文。

## 12. 安全落地

- 所有外部 API 必须走 API Gateway。
- API Client 最小权限和账户范围绑定。
- 供应商凭证用 KMS 加密。
- Webhook 使用 HMAC 签名。
- 登录和敏感操作支持 MFA。
- 敏感字段脱敏展示。
- 审计日志不可由普通管理员删除。
- 批量导出需要权限和审计。

## 13. 观测与运维

日志：

- request_id
- correlation_id
- account_id
- user_id/api_client_id
- supplier_id
- operation_id

指标：

- API p95/p99 latency。
- CDR backlog。
- Rating backlog。
- Billing run duration。
- Supplier API success rate。
- eSIM operation success rate。
- Queue lag。
- Webhook delivery success rate。

告警：

- CDR 延迟超过阈值。
- Supplier API 连续失败。
- Rating 积压。
- Billing 任务失败。
- 数据库连接池耗尽。
- 队列堆积。
- Webhook 大量失败。

## 14. 部署方案

推荐云端部署：

- Kubernetes 或云厂商容器服务。
- 多可用区。
- API Gateway + Load Balancer。
- 托管 PostgreSQL。
- 托管 Redis。
- 托管 Kafka/RabbitMQ。
- 对象存储。
- KMS。
- CI/CD 流水线。

环境：

- dev
- test
- staging
- production
- sandbox，给客户 API 联调使用。

发布策略：

- 蓝绿或滚动发布。
- 数据库迁移先兼容后清理。
- Supplier Adapter 独立灰度。
- 高风险 Billing 和 Rating 发布需要回归测试。

## 15. 模块拆分优先级

优先独立服务：

- Supplier Integration：外部变化大，故障隔离价值高。
- CDR Ingestion：吞吐高，处理链路长。
- Rating：规则复杂，测试要求高。
- Billing：财务严肃性高，边界清晰。
- eSIM Orchestration：状态机复杂，供应商依赖强。

可以晚些拆分：

- Settings。
- Audit。
- Notification。
- Dashboard Analytics。
