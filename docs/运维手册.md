# CMP 运维手册

## 1. 本地启动

在项目根目录运行：

```powershell
npm start
```

默认地址：

- 前端：`http://localhost:8080/`
- API：`http://localhost:8080/api/v1`
- 健康检查：`http://localhost:8080/api/v1/health`

如果本机没有 Node.js，可使用 Codex 工作区捆绑 Node：

```powershell
C:\Users\61004\.cache\codex-runtimes\codex-primary-runtime\dependencies\node\bin\node.exe backend\src\server.js
```

## 2. 环境变量

阶段一支持：

| 变量 | 默认值 | 说明 |
| --- | --- | --- |
| `PORT` | `8080` | 服务端口 |
| `DATABASE_URL` | 空 | 配置后启用 PostgreSQL JSONB 持久层 |
| `JWT_SECRET` | 开发默认值 | JWT HMAC 签名密钥，生产必须修改 |
| `JWT_TTL_SECONDS` | `28800` | Token 有效期 |
| `ALLOW_HEADER_AUTH` | `false` | 是否允许 `X-User-Id` 模拟登录，仅本地调试可开 |

生产环境建议增加 `DATABASE_URL`、`REDIS_URL`、`JWT_ISSUER`、`JWT_AUDIENCE`、`KMS_KEY_ID`、`OBJECT_STORAGE_BUCKET`、`SMTP_HOST`、`SUPPLIER_SECRET_*`。敏感配置必须存放在云密钥服务或 Secret Manager 中，不能写入代码库。

启用 PostgreSQL 前需要安装依赖：

```powershell
npm install
```

最小 PostgreSQL 启动示例：

```powershell
$env:DATABASE_URL="postgres://user:password@host:5432/localcmp"
$env:JWT_SECRET="<replace-with-random-secret>"
npm start
```

服务首次连接 PostgreSQL 时会创建 `runtime.collection_items`，并把 `backend/data/seed.json` 初始化进去。该持久层用于阶段一 MVP；后续可逐步替换为正式 repository 和领域表。

## 3. 健康检查

```powershell
Invoke-RestMethod http://localhost:8080/api/v1/health
Invoke-RestMethod http://localhost:8080/api/v1/dashboard/overview
Invoke-RestMethod http://localhost:8080/api/v1/sims
Invoke-RestMethod http://localhost:8080/api/v1/openapi
```

生产环境应区分 liveness、readiness 和 startup 检查。

本地核心链路冒烟：

```powershell
npm run smoke
```

该命令会启动临时服务并验证 health、auth context、SIM 预览、CDR 导入、批量任务、eSIM 操作查询和 API Client 查询。

## 4. 日志与追踪

每个请求应携带或生成 `X-Correlation-Id`，并贯穿 API 请求日志、audit logs、outbox events、supplier api logs、batch job item 和 invoice delivery。

日志字段建议：timestamp、level、service、route、account_id、user_id、correlation_id、request_id、latency_ms、status_code、error_code。

## 5. 指标与告警

核心指标：

- API p95/p99 latency。
- HTTP 4xx/5xx 比例。
- 数据库连接池使用率。
- CDR backlog 和延迟分钟数。
- Rating backlog。
- Billing run duration。
- Supplier API success rate。
- eSIM operation success rate。
- Batch job failure rate。
- Webhook delivery success rate。

告警建议：API 延迟超 SLA、CDR 延迟超过 30 分钟、供应商连续失败、Billing run 失败、队列积压、数据库连接池耗尽、审计日志写入失败。

## 6. 数据库运维

PostgreSQL 建议：

- 核心业务表按 schema 拆分。
- CDR、rated_usage、audit_logs、supplier_api_logs 按月分区。
- 所有外键列建立索引。
- 高并发查询建立组合索引，例如 `(account_id, status)`、`(sim_id, event_time)`。
- 开启 RLS 保护多租户数据边界。
- 使用迁移工具管理 DDL，禁止生产手工改表。

备份要求：每日全量备份、持续 WAL 归档、每月恢复演练、对象存储开启版本控制。

## 7. 发布流程

1. CI 执行语法检查、单元测试、API 合约测试。
2. 数据库迁移先做兼容变更。
3. Staging 回放 SIM、CDR、Billing、批量任务流程。
4. 生产滚动发布或蓝绿发布。
5. 观察错误率、延迟、队列积压、供应商成功率。
6. 确认稳定后清理旧字段或旧逻辑。

高风险模块如 Billing、Rating、Supplier Adapter、eSIM Orchestration 必须支持灰度和快速回滚。

阶段一发布前最低检查：

```powershell
npm run check
npm run smoke
```

如果仅修改前端静态资源，也至少执行 `npm run check` 并手动打开 Dashboard、SIM、用量、Billing、批量任务、API 与 Webhook 页面。

## 8. 故障处理

CDR 延迟：查看供应商健康、同步时间、导入任务、parse error、duplicate 数量，必要时触发重拉或重跑。

SIM 操作长时间 processing：查询 operation、outbox event、供应商 transaction_id 和 API log；供应商结果 unknown 时进入补偿查询，禁止直接改最终状态。

账单发送失败：查看 invoice 状态、通知队列、邮件服务日志和账单联系人邮箱，多次失败后人工发送并保留审计。

Webhook 投递失败：查看 `notification-deliveries`、目标地址 TLS 证书、HTTP 状态码、超时、签名校验错误和重试次数。连续失败超过阈值时暂停订阅并通知客户开发者。

API Client 异常：查看调用 IP、Scope、限流命中、最近轮换时间和审计日志。疑似泄露时立即 `suspend`，创建新 Client 并通知客户切换。

## 9. 安全基线

- 生产必须使用 HTTPS。
- 后台登录启用 MFA。
- 生产必须设置强随机 `JWT_SECRET`。
- 生产不要开启 `ALLOW_HEADER_AUTH`。
- API Client 使用最小 Scope。
- 供应商凭证使用 KMS 加密。
- Webhook 使用 HMAC 签名。
- API Client Secret 和 Webhook Signing Secret 只返回一次，服务端只保存 hash。
- CDR 和账单导出需要权限和审计。
- 敏感字段默认脱敏展示。
- 审计日志不可由普通管理员删除。
